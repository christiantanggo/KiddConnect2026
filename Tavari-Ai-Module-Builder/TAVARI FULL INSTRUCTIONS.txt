GAP #2: ACTIVE BUSINESS CONTEXT ENFORCEMENT AT API LEVEL

Problem: UI behavior is defined, but server-side enforcement is missing.
Users could access data from other businesses if only client-side enforcement
exists.

SOLUTION: requireBusinessContext Middleware

Every API request MUST:

1. Resolve active_business_id from:
   - Request header: X-Active-Business-Id
   - Session: active_organization_id
   - Fallback: user's primary business_id (if only one organization)

2. Verify user belongs to that business:
   - Check organization_users table for (user_id, business_id) match
   - Verify deleted_at IS NULL
   - Reject with 403 Forbidden if user doesn't belong

CRITICAL: Always query organization_users table for authorization.
DO NOT use users.business_id for authorization checks - it is only a fallback
for convenience when determining which business to auto-select.

3. Attach business context to request:
   - req.active_business_id
   - req.business (full business object)

4. Reject requests without valid business context:
   - 400 Bad Request if no business_id can be determined
   - 403 Forbidden if user doesn't belong to business

Implementation:
- Apply to ALL module APIs (e.g., /api/{module_key}/*)
- Apply to ALL settings APIs (/api/settings/*)
- Apply to ALL billing APIs (/api/billing/*)
- Skip only for auth endpoints (/api/auth/*)

Example Middleware:
requireBusinessContext(req, res, next) {
  const activeBusinessId = 
    req.headers['x-active-business-id'] ||
    req.session.active_organization_id ||
    (req.user.organization_count === 1 ? req.user.primary_business_id : null);
    
  if (!activeBusinessId) {
    return res.status(400).json({ error: 'Active business context required' });
  }
  
  // CRITICAL: Verify user belongs to business via organization_users
  // DO NOT check users.business_id - it's not for authorization
  const membership = await organization_users.findOne({
    user_id: req.user.id,
    business_id: activeBusinessId,
    deleted_at: null
  });
  
  if (!membership) {
    return res.status(403).json({ error: 'Access denied to this business' });
  }
  
  req.active_business_id = activeBusinessId;
  req.business_membership = membership;
  next();
}

RISK #2: API RATE LIMITING (AI COST CONTROL)

Problem: Usage limits track overall consumption, but don't prevent prompt abuse,
scripted attacks, or cost spikes before usage logs catch up.

SOLUTION: Request-Level Rate Limiting

All AI generate endpoints MUST implement:
- Per-user rate limiting (e.g., X requests per minute per user)
- Per-business rate limiting (e.g., Y requests per minute per business)
- Per-IP rate limiting (optional, for additional protection)

Rate limits are SEPARATE from usage limits:
- Rate limits: Prevent abuse and cost spikes (requests per time period)
- Usage limits: Track consumption for billing (total units used)

Implementation:
- Use rate limiting middleware (e.g., express-rate-limit)
- Configure different limits per module if needed
- Store rate limit state in Redis or similar (fast, shared across instances)
- Return 429 Too Many Requests when limit exceeded
- Log rate limit violations in audit_logs

Example:
// Apply to all AI generation endpoints
const aiRateLimiter = rateLimit({
  windowMs: 60 * 1000, // 1 minute
  max: async (req) => {
    // Per-user limit
    return 10; // 10 requests per minute per user
  },
  standardHeaders: true,
  legacyHeaders: false,
  handler: (req, res) => {
    // Log violation
    audit_logs.create({
      action: 'rate_limit_exceeded',
      user_id: req.user.id,
      business_id: req.active_business_id,
      metadata: { endpoint: req.path }
    });
    res.status(429).json({
      error: 'Too many requests. Please try again later.'
    });
  }
});

// Apply to routes:
router.post('/api/:moduleKey/generate',
  requireBusinessContext,
  requireLegalAcceptance,
  aiRateLimiter, // Rate limiting BEFORE AI processing
  verifySubscription,
  checkUsageLimits,
  generateAI
);

Rate Limit Configuration:
- Define limits in environment variables or config file
- Default: 10 requests/minute per user, 100 requests/minute per business
- Can be adjusted per module based on expected usage patterns
- Should be documented but exact numbers can be tuned in production

GAP #3: SUBSCRIPTION STATUS SOURCE OF TRUTH

Problem: Both Stripe subscription status and subscriptions.status exist, but
there's no clear rule for which wins when they disagree. This will happen due
to webhook delays, failed renewals, etc.

SOLUTION: Stripe is Source of Truth

Rule:
- Stripe subscription status is the SOURCE OF TRUTH
- subscriptions.status is a CACHED MIRROR for performance
- subscriptions.status must be kept in sync via webhooks
- UI/API should verify with Stripe API for critical operations

Sync Logic:

1. Stripe Webhooks Update subscriptions.status:
   - customer.subscription.updated → Update subscriptions.status
   - customer.subscription.deleted → Update subscriptions.status = 'canceled'
   - invoice.payment_succeeded → Update subscriptions.status = 'active'
   - invoice.payment_failed → Update subscriptions.status = 'past_due' (if applicable)

CRITICAL: All Stripe and ClickBank webhooks MUST verify provider signatures
before processing. This prevents webhook spoofing and ensures data integrity.

Webhook Signature Verification:
- Stripe: Verify signature using Stripe webhook secret
- ClickBank: Verify signature using ClickBank webhook secret
- Reject webhooks with invalid signatures (return 401 Unauthorized)
- Log signature verification failures in audit_logs

Implementation:
// Stripe webhook verification
app.post('/api/webhooks/stripe', express.raw({type: 'application/json'}), (req, res) => {
  const sig = req.headers['stripe-signature'];
  let event;
  
  try {
    event = stripe.webhooks.constructEvent(
      req.body,
      sig,
      process.env.STRIPE_WEBHOOK_SECRET
    );
  } catch (err) {
    // Invalid signature
    audit_logs.create({
      action: 'webhook_signature_invalid',
      metadata: { provider: 'stripe', error: err.message }
    });
    return res.status(401).send('Webhook signature verification failed');
  }
  
  // Process verified webhook
  handleStripeWebhook(event);
  res.json({received: true});
});

// ClickBank webhook verification
app.post('/api/webhooks/clickbank', (req, res) => {
  const signature = req.headers['x-clickbank-signature'];
  const payload = JSON.stringify(req.body);
  
  if (!verifyClickBankSignature(payload, signature)) {
    audit_logs.create({
      action: 'webhook_signature_invalid',
      metadata: { provider: 'clickbank' }
    });
    return res.status(401).send('Webhook signature verification failed');
  }
  
  // Process verified webhook
  handleClickBankWebhook(req.body);
  res.json({received: true});
});

2. Before Critical Operations, Verify with Stripe:
   - Before allowing module access, check Stripe API
   - Before processing payment, check Stripe API
   - Cache result for short period (5-15 minutes) to avoid rate limits

3. Background Sync Job:
   - Periodic job (every 6 hours) to sync subscription statuses
   - Reconciles discrepancies between Stripe and database
   - Logs discrepancies in audit_logs for investigation

Implementation Pattern:
async function verifySubscriptionStatus(businessId, moduleKey) {
  // Check cache first (subscriptions table)
  const subscription = await subscriptions.findOne({
    business_id: businessId,
    module_key: moduleKey
  });
  
  // For critical operations, verify with Stripe
  if (subscription && subscription.stripe_subscription_item_id) {
    const stripeItem = await stripe.subscriptionItems.retrieve(
      subscription.stripe_subscription_item_id
    );
    const stripeSubscription = await stripe.subscriptions.retrieve(
      stripeItem.subscription
    );
    
    // Update cache if different
    if (stripeSubscription.status !== subscription.status) {
      await subscriptions.update(subscription.id, {
        status: stripeSubscription.status
      });
      // Log discrepancy
      await audit_logs.create({
        action: 'subscription_status_sync',
        business_id: businessId,
        metadata: {
          cached_status: subscription.status,
          stripe_status: stripeSubscription.status
        }
      });
    }
    
    return stripeSubscription.status === 'active';
  }
  
  return false;
}

RISK #3: PARTIAL STRIPE FAILURE STATE

Problem: Edge case where Stripe operation succeeds but database update fails
(network issue, crash, etc.), leaving system in inconsistent state.

Example:
- Stripe subscription item removed successfully
- Database update fails (network error, crash)
- Result: Stripe shows item removed, DB shows it still active
- User loses access but system thinks they still have it

SOLUTION: Idempotent Operations with Retry Logic

All Stripe-mutating operations must be idempotent and retry-safe.

Rule:
- All Stripe operations must be idempotent (safe to retry)
- Failed DB updates after Stripe operations must be retried via background job
- Use idempotency keys for Stripe operations where possible
- Track operation state in database (e.g., pending, completed, failed)

Implementation Pattern:
async function removeStripeSubscriptionItem(subscriptionId, itemId) {
  // Use idempotency key to prevent duplicate operations
  const idempotencyKey = `remove_item_${itemId}_${Date.now()}`;
  
  try {
    // Step 1: Remove from Stripe
    await stripe.subscriptionItems.del(itemId, {
      idempotency_key: idempotencyKey
    });
    
    // Step 2: Update database
    await subscriptions.update(subscriptionId, {
      status: 'canceled',
      stripe_subscription_item_id: null
    });
    
    // Success
    return { success: true };
    
  } catch (error) {
    // If Stripe succeeded but DB failed, queue retry
    if (error.type === 'database_error' || error.code === 'ECONNREFUSED') {
      // Queue background job to retry DB update
      await backgroundJobs.queue({
        type: 'sync_stripe_to_db',
        stripe_item_id: itemId,
        subscription_id: subscriptionId,
        operation: 'remove',
        retry_count: 0,
        max_retries: 5
      });
      
      // Log for monitoring
      await audit_logs.create({
        action: 'stripe_db_sync_queued',
        business_id: subscription.business_id,
        metadata: {
          stripe_item_id: itemId,
          error: error.message
        }
      });
    }
    
    throw error;
  }
}

Background Retry Job:
// Runs periodically (every 5 minutes)
async function retryStripeSyncJobs() {
  const failedJobs = await backgroundJobs.find({
    type: 'sync_stripe_to_db',
    status: 'pending',
    retry_count: { $lt: 5 }
  });
  
  for (const job of failedJobs) {
    try {
      // Verify Stripe state
      const stripeItem = await stripe.subscriptionItems.retrieve(job.stripe_item_id);
      
      // Update DB to match Stripe
      if (job.operation === 'remove' && !stripeItem) {
        // Item removed in Stripe, update DB
        await subscriptions.update(job.subscription_id, {
          status: 'canceled',
          stripe_subscription_item_id: null
        });
        
        // Mark job complete
        await backgroundJobs.update(job.id, { status: 'completed' });
        
      } else if (job.operation === 'add' && stripeItem) {
        // Item exists in Stripe, update DB
        await subscriptions.update(job.subscription_id, {
          stripe_subscription_item_id: stripeItem.id,
          status: 'active'
        });
        
        // Mark job complete
        await backgroundJobs.update(job.id, { status: 'completed' });
      }
      
    } catch (error) {
      // Increment retry count
      await backgroundJobs.update(job.id, {
        retry_count: job.retry_count + 1,
        last_error: error.message
      });
      
      // If max retries reached, alert admin
      if (job.retry_count >= 4) {
        await notifications.create({
          business_id: job.business_id,
          type: 'warning',
          message: 'Subscription sync failed. Manual intervention required.'
        });
      }
    }
  }
}

GAP #4: CLICKBANK REFUND HANDLING

Problem: ClickBank refunds must automatically revoke Stripe subscription items,
otherwise paid access remains active after refund, creating support nightmares.

SOLUTION: Automated Refund Webhook Handler

ClickBank Refund Webhook Flow:

1. Verify webhook signature (CRITICAL - see webhook verification above)

2. Receive refund webhook from ClickBank

3. Update external_purchases:
   - Find by external_order_id
   - Update status = 'refunded'
   - Store refund data in purchase_data JSONB

4. Remove Stripe Subscription Item (with retry logic):
   - Find subscription by business_id + module_key
   - Get stripe_subscription_item_id
   - Call: stripe.subscriptionItems.del(stripe_subscription_item_id) (idempotent)
   - If Stripe succeeds but DB update fails, queue retry job

5. Update subscriptions table:
   - Update subscriptions.status = 'canceled'
   - Update subscriptions.ends_at = NOW()
   - Set usage_limit = 0 (immediately block access)

6. Log in audit_logs:
   - action: 'clickbank_refund_processed'
   - business_id, module_key
   - external_order_id, refund_amount
   - metadata: full refund details

7. Send notification:
   - Create notification for business
   - Type: 'billing'
   - Message: "Your {module_name} subscription has been canceled due to refund"

Implementation:
POST /api/webhooks/clickbank/refund

async function handleClickBankRefund(webhookData) {
  // Verify signature FIRST
  if (!verifyClickBankSignature(webhookData)) {
    return res.status(401).send('Invalid signature');
  }
  
  const { order_id, refund_amount } = webhookData;
  
  // Find purchase
  const purchase = await external_purchases.findOne({
    provider: 'clickbank',
    external_order_id: order_id
  });
  
  if (!purchase || purchase.status === 'refunded') {
    return; // Already processed or not found
  }
  
  // Update purchase
  await external_purchases.update(purchase.id, {
    status: 'refunded',
    purchase_data: { ...purchase.purchase_data, refund: webhookData }
  });
  
  // Find subscription
  const subscription = await subscriptions.findOne({
    business_id: purchase.business_id,
    module_key: purchase.module_key
  });
  
  if (subscription && subscription.stripe_subscription_item_id) {
    try {
      // Remove Stripe subscription item (idempotent operation)
      await removeStripeSubscriptionItem(
        subscription.id,
        subscription.stripe_subscription_item_id
      );
    } catch (error) {
      // If removal fails, queue retry
      // (Error handling already in removeStripeSubscriptionItem function)
    }
  }
  
  // Update subscription status
  await subscriptions.update(subscription.id, {
    status: 'canceled',
    ends_at: new Date(),
    usage_limit: 0
  });
  
  // Log
  await audit_logs.create({
    action: 'clickbank_refund_processed',
    business_id: purchase.business_id,
    metadata: {
      module_key: purchase.module_key,
      external_order_id: order_id,
      refund_amount
    }
  });
  
  // Notify
  await notifications.create({
    business_id: purchase.business_id,

ADDITION #2: SUPPORT IMPERSONATION CONTRACT

Problem: Admin access is mentioned, but impersonation capabilities aren't
explicitly defined. This will save massive support time.

SOLUTION: Admin Impersonation System

Rules:

1. Tavari Admin Can Impersonate Users:
   - Only admin_users with role 'admin' or 'support' can impersonate
   - Can impersonate in two modes:
     a. Read-only mode: View user's data, cannot make changes
     b. Full mode: Act as user, make changes on their behalf

2. Impersonation Process:
   - Admin initiates via: POST /admin/impersonate
   - Parameters: user_id, mode ('readonly' | 'full')
   - System creates impersonation session
   - Admin's requests include X-Impersonating-User-Id header
   - Original admin_user_id stored in req.impersonated_by

3. Audit Logging:
   - Log impersonation_start action in audit_logs
   - Log all actions during impersonation with:
     - user_id: Impersonated user's ID
     - admin_user_id: Admin who is impersonating
     - action: 'impersonation_start', 'impersonation_end'
     - metadata: { mode, reason, duration }
   - Log impersonation_end when session ends

4. UI Indicators:
   - Show banner: "Impersonating: [User Name] | Mode: [Read-only/Full] | End Session"
   - All pages show impersonation context
   - All actions show in audit log under impersonated user but with admin_user_id

5. Restrictions:
   - Cannot impersonate other admin_users
   - Cannot access /admin routes while impersonating (except /admin/impersonate/end)
   - Read-only mode: All POST/PUT/DELETE requests return 403
   - Session expires after 2 hours (configurable)

Implementation:
// Start impersonation
POST /admin/impersonate
Body: { user_id, mode: 'readonly' | 'full', reason }

async function startImpersonation(req, res) {
  const adminUser = req.admin_user;
  const { user_id, mode, reason } = req.body;
  
  // Verify target user is not an admin
  const targetUser = await users.findById(user_id);
  if (!targetUser || targetUser.is_admin) {
    return res.status(403).json({ error: 'Cannot impersonate this user' });
  }
  
  // Create impersonation session
  req.session.impersonation = {
    user_id,
    mode,
    started_at: new Date(),
    admin_user_id: adminUser.id,
    reason
  };
  
  // Log
  await audit_logs.create({
    action: 'impersonation_start',
    admin_user_id: adminUser.id,
    user_id,
    metadata: { mode, reason }
  });
  
  res.json({ 
    success: true,
    impersonating: user_id,
    mode 
  });
}

// Middleware to apply impersonation context
async function applyImpersonation(req, res, next) {
  if (req.session.impersonation) {
    const { user_id, mode, admin_user_id } = req.session.impersonation;
    
    // Override user context
    req.original_user = req.user;
    req.user = await users.findById(user_id);
    req.impersonated_by = admin_user_id;
    req.impersonation_mode = mode;
    
    // In read-only mode, block mutations
    if (mode === 'readonly' && ['POST', 'PUT', 'DELETE', 'PATCH'].includes(req.method)) {
      return res.status(403).json({
        error: 'Read-only impersonation mode. Cannot make changes.'
      });
    }
    
    // Attach to audit log context
    req.audit_context = {
      ...req.audit_context,
      impersonated: true,
      impersonated_by: admin_user_id,
      impersonation_mode: mode
    };
  }
  
  next();
}

// End impersonation
POST /admin/impersonate/end

async function endImpersonation(req, res) {
  const impersonation = req.session.impersonation;
  
  if (impersonation) {
    const duration = Date.now() - new Date(impersonation.started_at);
    
    await audit_logs.create({
      action: 'impersonation_end',
      admin_user_id: impersonation.admin_user_id,
      user_id: impersonation.user_id,
      metadata: {
        mode: impersonation.mode,
        duration_ms: duration,
        reason: impersonation.reason
      }
    });
    
    delete req.session.impersonation;
  }
  
  res.json({ success: true });
}

================================================================================
UNIFIED BILLING ARCHITECTURE
================================================================================

Principle: ONE Stripe System, Multiple Module Entitlements

businesses table:
  stripe_customer_id: "cus_xxxxx"        ← ONE customer per business
  stripe_subscription_id: "sub_xxxxx"    ← ONE subscription per business

Stripe Subscription (sub_xxxxx) contains multiple items:
  - Subscription Item 1: Phone Agent (price_id: price_phone)
  - Subscription Item 2: Sales AI (price_id: price_sales) → "si_xxxxx"
  - Subscription Item 3: Content AI (price_id: price_content) → "si_yyyyy"

subscriptions table:
  business_id + 'phone' → No item_id (uses businesses.stripe_subscription_id directly)
  business_id + 'sales' → stripe_subscription_item_id: "si_xxxxx"
  business_id + 'content' → stripe_subscription_item_id: "si_yyyyy"

Billing Logic:

For Phone Agent:
- Check businesses.stripe_subscription_id status directly
- Use existing logic

For v2 Modules:
1. Check businesses.stripe_subscription_id status (must be active)
2. Check subscriptions table for module entitlement
3. Check subscriptions.status = 'active' (but verify with Stripe for critical ops)
4. Check subscriptions.usage_limit vs actual usage
5. Use subscriptions.stripe_subscription_item_id for Stripe operations

CRITICAL: Always verify subscription status with Stripe API for critical
operations. subscriptions.status is cached for performance but Stripe is
source of truth.

Adding a Module to Subscription:

When activating a module:
1. Get or create subscription: businesses.stripe_subscription_id
2. Add subscription item: stripe.subscriptions.update(subscription_id, {
     items: [{ price: module.stripe_price_id_monthly }]
   })
3. Save returned subscription_item.id to subscriptions.stripe_subscription_item_id

CRITICAL: All Stripe operations must be idempotent and retry-safe. If Stripe
succeeds but database update fails, queue background job to retry DB update.

ClickBank Refund Handling:

When ClickBank refund webhook received:
1. Verify webhook signature FIRST (CRITICAL)
2. Update external_purchases.status = 'refunded'
3. Remove Stripe subscription item immediately (idempotent operation)
4. Update subscriptions.status = 'canceled' (with retry if DB fails)
5. Set subscriptions.usage_limit = 0 (immediately block access)
6. Log in audit_logs
7. Send notification to business

If Stripe removal succeeds but DB update fails, background job will retry
until successful.

- Deactivation immediately revokes access
- Refunds revoke access automatically (ClickBank webhook)

Subscription Status Verification:

CRITICAL RULE: Stripe is source of truth, subscriptions.status is cached mirror.

For critical operations (module access, payment processing):
1. Check subscriptions.status (cached, fast)
2. Verify with Stripe API (source of truth, slower)
3. Update subscriptions.status if different
4. Log discrepancies in audit_logs

For non-critical operations (UI display, dashboards):
- Can use cached subscriptions.status
- Background sync job keeps it updated

CRITICAL: All Stripe operations must be idempotent and retry-safe. If Stripe
succeeds but DB update fails, background job will retry DB update until successful.

Usage Tracking:

- Units: Each AI action consumes units (configurable per module)
- Logging: All usage logged to usage_logs table
- Limits: Enforced per module/plan combination (in subscriptions.usage_limit)
- Blocking: If usage exceeds limit → block action, show upgrade modal

Rate Limiting (Separate from Usage Limits):

- Per-user rate limiting: X requests per minute (prevents prompt abuse)
- Per-business rate limiting: Y requests per minute (prevents cost spikes)
- Per-IP rate limiting: Optional additional protection
- Rate limits are SEPARATE from usage limits
- Rate limits: Prevent abuse and cost spikes (requests per time period)
- Usage limits: Track consumption for billing (total units used)

Universal Rule:
IF businesses.stripe_subscription_id IS ACTIVE (verify with Stripe)
  AND subscriptions.status = 'active' (cached)
  AND module.health_status = 'healthy'
  AND usage < subscriptions.usage_limit 
  AND rate limit not exceeded
→ allow
ELSE → show upgrade modal or appropriate error

ClickBank Integration:

- ClickBank purchases tracked in external_purchases table
- Webhook handler creates subscriptions record
- Links to existing businesses record via email lookup
- Creates business if doesn't exist
- Adds subscription item to unified Stripe subscription

ClickBank Refund Handling (CRITICAL):

When refund webhook received:
1. Verify webhook signature FIRST (CRITICAL - reject if invalid)
2. Update external_purchases.status = 'refunded'
3. Remove Stripe subscription item immediately (idempotent operation)
4. Update subscriptions.status = 'canceled' (with retry if DB fails)
5. Set subscriptions.usage_limit = 0
6. Log in audit_logs
7. Send notification to business

If Stripe removal succeeds but DB update fails, background job will retry
until successful.
